# Flow 개발 방향: AI Agent 중심 운영 모델

이 문서는 Flow를 단순 CLI 모음이 아니라, 스펙을 중심으로 여러 Agent가 협업하는 개발 운영 시스템으로 재정의하기 위한 방향 메모다.

핵심 전제는 다음과 같다.

- 사용자는 채팅 또는 웹 UI에서 요구사항을 제시하고 검토한다.
- Agent는 스펙 작성, 구현, 테스트, 검증, 상태 변경을 역할별로 분리해서 수행한다.
- 모든 판단 근거와 산출물은 스펙 단위로 추적 가능해야 한다.
- 하나의 저장소만이 아니라 여러 프로젝트를 동일한 운영 모델로 다룰 수 있어야 한다.

이 문서는 제품 방향과 운영 모델을 정리한 문서다. 기존 코드와 문서는 참고 자료로만 사용하고, 실제 구현은 이 방향을 기준으로 새로 작성하는 것을 전제로 한다.

# 1. 왜 구조를 바꾸는가

현재 Flow는 CLI, Runner, 스펙 그래프, E2E 테스트, VS Code 확장 등 여러 축이 이미 존재한다. 하지만 앞으로는 기능을 추가하는 수준이 아니라, 아래 관점으로 시스템 중심축을 바꾸는 편이 맞다.

- 중심 객체를 명령이 아니라 스펙으로 둔다.
- 실행 주체를 단일 CLI 사용자가 아니라 역할별 Agent로 둔다.
- 작업 완료 기준을 코드 작성이 아니라 acceptance criteria 검증과 증거 축적으로 둔다.
- 저장소별 로컬 도구가 아니라 멀티 프로젝트 운영 플랫폼으로 확장한다.

즉 Flow는 "빌드/테스트 도구"라기보다 "스펙 기반 AI 개발 운영 시스템"이 되어야 한다.

# 2. 목표

- 여러 프로젝트를 공통된 스펙 구조와 상태 모델로 관리한다.
- Planner, Developer, Validator 같은 Agent가 명확히 역할 분리된 파이프라인으로 협업한다.
- 스펙, 테스트, 증거, 상태 전이, 사용자 피드백을 한곳에서 추적한다.
- 웹 서비스와 CLI가 같은 저장 구조와 같은 상태 모델을 사용한다.
- 사람이 중간에 개입해도 전체 이력과 의사결정이 끊기지 않게 만든다.

# 3. 멀티 프로젝트 저장 구조

여러 저장소를 지원하려면 Flow 홈 디렉터리 아래에 프로젝트별 작업 공간이 필요하다.

```text
~/.flow/
   projects/
      <project-id>/
         config.yaml
         specs/
         evidence/
         runs/
         artifacts/
         logs/
         cache/
```

저장 구조 원칙은 다음과 같이 고정한다.

- `<project-id>`는 폴더명이 아니라 안정적인 식별자를 사용한다.
- 워크스페이스 루트 경로, 원격 저장소 URL, 표시 이름을 `config.yaml`에 함께 둔다.
- 스펙, 실행 로그, evidence는 모두 shared home에 저장한다.
- 프로젝트별 설정과 전역 설정을 분리한다.
- 호환 계층은 두지 않는다. 새 구조는 `~/.flow/projects/<project-id>/...`를 기준으로 바로 간다.
- 저장소 내부 `docs/evidence` 같은 경로는 사용하지 않는다.

# 4. 핵심 운영 개념

## 4.1 단위

Flow의 최소 운영 단위는 스펙이다. Agent는 브랜치나 PR 단위가 아니라 스펙 단위로 일한다.

스펙은 최소한 아래를 포함해야 한다.

- 무엇을 바꾸는지에 대한 정의
- 왜 필요한지에 대한 배경과 목적
- problem
- goal
- acceptance criteria
- acceptance criteria와 연결된 테스트
- risk level
- 현재 상태와 처리 상태
- 의존성
- 활동 이력과 근거 산출물
- 검토 요청과 사용자 응답 이력
- assignment 이력

## 4.2 SoT 원칙

- 구조화된 JSON 스펙이 authoritative source다.
- 사람이 읽는 문서 화면은 JSON 스펙을 가공해 보여주는 view여야 한다.
- 코드가 먼저 바뀔 수는 있지만, 스펙 없는 완료는 허용하지 않는다.
- 테스트는 acceptance criteria를 검증하기 위한 수단이어야 한다.
- 상태 전이와 판정은 로그와 증거를 남겨야 한다.

즉 `spec.md` 같은 사람이 편집하는 원본 문서는 두지 않는다. 사람이 보는 스펙 편집기나 문서 뷰는 모두 JSON 스펙을 기반으로 생성되거나 편집되어야 한다.

## 4.3 사용자 개입 원칙

- 사용자는 최종 승인자이자 요구사항 조정자다.
- Agent는 사용자가 답해야 할 질문과 검토 포인트를 구조화해서 올린다.
- 사용자의 답변, 수동 테스트 결과, 승인 여부는 모두 스펙 이력에 남긴다.

# 5. Agent 역할 정의

역할을 겹치게 두면 판단 책임이 흐려진다. 아래처럼 책임을 나누는 편이 적절하다.

## 5.1 Planner

권장 구현: Claude CLI 또는 대화형 Agent

책임:

- 사용자와 직접 소통
- 요구사항 해석
- 스펙 범위 분석
- 새 스펙 생성 또는 기존 스펙 변경
- 열린 질문 정리
- 우선순위 및 의존성 조정

Planner는 구현을 직접 완료하는 역할이 아니라, 일을 수행 가능한 스펙으로 정리하는 역할이다.

Planner가 작성한 초안은 `대기`로 올리기 전에 Spec Validator가 acceptance criteria의 테스트 가능성을 한 번 점검한다. 이 프리패스는 무거운 설계 리뷰가 아니라, "이 기준이 실제 테스트와 증거로 검증 가능한가"를 빠르게 확인하는 단계다.

## 5.2 Architect

권장 구현: Claude CLI

책임:

- 여러 스펙에 걸친 구조 영향 분석
- 시스템 경계와 모듈 책임 조율
- 구현 전략 초안 정리
- `구현 검토` 상태에서 구현 착수 전 아키텍처 검토
- 테스트 전략 가이드 제공
- 위험도가 큰 변경의 설계 리뷰

주의할 점:

- Architect가 직접 구현까지 가져가면 Developer와 경계가 흐려진다.
- 테스트 코드를 Architect가 항상 작성하는 것으로 고정하기보다, "테스트 전략과 구조를 책임진다"로 두는 편이 안전하다.
- Architect는 상태를 변경하지 않는다. 구현 착수 가능 여부에 대한 검토 결과와 권고만 남긴다.
- Architect의 구현 검토가 병목이 되지 않도록 체크리스트 기반으로 제한. 구조, 경계, 위험, 테스트 전략 정도
- AC 테스트 가능성 프리패스는 Spec Validator가 담당한다. Architect는 `구현 검토` 단계에서만 관여한다.

Risk Level 기반 권장 규칙:

- `low`: Architect 단계를 자동 통과하거나 생략 가능
- `medium`: 체크리스트 기반 경량 검토
- `high`: Architect 검토 필수
- `critical`: Architect 검토와 명시적 승인 필수

예시:

- 단순 UI 문구 수정, 사소한 화면 배치 조정: `low`
- 단일 기능 보강, 외부 계약 변화가 작은 수정: `medium`
- DB 스키마 변경, 공용 계약 변경, 모듈 경계 변경: `high`
- 데이터 마이그레이션, 런타임 구조 재편, 여러 프로젝트에 영향 주는 변경: `critical`

## 5.3 Developer

권장 구현: Copilot CLI

책임:

- 스펙 단위 코드 수정
- 필요한 테스트 추가 또는 수정
- 자신이 구현한 변경에 대한 빌드/테스트 실행
- 실패 로그와 시도 내역 기록
- 테스트 결과와 산출물 수집

Developer의 목표는 단순 구현이 아니라, 스펙의 acceptance criteria를 만족시키는 변경과 검증 재료를 만드는 것이다.

Developer는 상태를 변경하지 않는다. 구현 완료, 테스트 실행 완료, 실패, 보류 같은 작업 결과만 남기고 다음 검증과 상태 반영은 Test Validator, Spec Validator, Spec Manager가 역할에 따라 이어받는다.

## 5.4 Test Validator

권장 구현: Claude CLI

책임:

- 테스트가 acceptance criteria를 실제로 검증하는지 확인
- 부실한 테스트, 비결정적 테스트, 잘못된 검증 연결 식별
- 테스트 세트가 충분한지 판단
- 테스트 결과의 합격 여부를 최종 판정

이 역할은 단순히 "테스트가 실행되었는가"를 보는 것이 아니라, "이 테스트로 통과를 인정해도 되는가"와 "최종 합격으로 판정할 수 있는가"를 함께 본다.

## 5.5 Spec Validator

권장 구현: Claude CLI

책임:

- `초안` 상태 스펙의 acceptance criteria와 필수 필드의 형식적 완결성 및 테스트 가능성 사전 검토 (AC 프리패스)
- 처리 상태가 `검토`인 스펙을 검증
- 스펙 담당자의 작업 내역과 스펙 상세를 검토
- 사용자에게 검토가 필요한 부분이 있는지 판단
- 필요한 경우 검토 의견을 작성하고 처리 상태를 `사용자검토`로 변경
- 사용자가 검토를 완료하면 다시 처리 상태가 `검토`로 돌아온 스펙을 재검토
- 스펙 처리가 충분하면 처리 상태를 `완료`로 변경
- 스펙 처리가 불완전하면 처리 상태를 `대기`로 되돌려 재작업 루프로 보냄
- 재시도 한계를 넘기면 처리 상태를 `실패`로 변경
- 실패 시 검토 의견을 남기고, 폐기, 분할, 재시도 같은 후속 선택지를 제안
- 활동 이력 정리
- 스펙 메타데이터 유지

중요한 규칙은 아래와 같다.

- Spec Validator는 스펙의 처리 상태를 검증 관점에서 제어한다.
- 사용자 검토가 필요한 경우 반드시 검토 의견을 남기고 `사용자검토`로 보낸다.
- 처리 상태가 `실패`가 되면 그 단계에서 멈추고, 검토 의견을 통해 후속 방향을 제안한다.
- 사용자가 스펙을 다시 `초안`으로 되돌리기로 하면 Planner가 기존 내역을 검토하고 스펙 분할 또는 변경을 수행한다.

Spec Validator의 처리 순서는 아래처럼 고정하는 편이 맞다.

1. 처리 상태가 `검토`인 스펙을 읽는다.
2. 담당자 작업 내역, assignment 결과, 스펙 상세를 확인한다.
3. 사용자 판단이 필요하면 검토 의견을 작성하고 처리 상태를 `사용자검토`로 바꾼다.
4. 사용자 검토 완료 후 다시 `검토`로 돌아오면 재판정한다.
5. 적합하면 처리 상태를 `완료`로 바꾼다.
6. 부적합하면 처리 상태를 `대기`로 되돌린다.
7. 같은 루프가 3회 반복되어도 수렴하지 않으면 처리 상태를 `실패`로 바꾼다.

Developer와 Test Validator의 판정, Architect 검토 결과, 사용자 응답 사이에 충돌이 생기면 Spec Validator가 retry loop를 관리한다.

- 자동 재시도 루프는 최대 3회까지 허용한다.
- 3회 이후에도 부적합이면 `실패`로 처리한다.
- `실패` 상태에서는 검토 의견에 폐기, 스펙 분할, 재시도 같은 대안을 남긴다.

## 5.6 Spec Manager

권장 구현: Claude CLI

책임:

- 스펙 상태 변경의 유일한 권한자
- Spec Validator 이후의 기계적 검토를 담당
- 처리 상태가 `완료`인 스펙에 대해 상태 흐름표를 적용
- 상태 흐름표에 따라 다음 상태를 결정하고 반영
- 상태 전이와 관련된 메타데이터를 기록
- 처리 상태가 `실패`, `보류`, `사용자검토`인 스펙은 상태를 진행시키지 않음

중요한 규칙은 하나다.

- 스펙 상태는 Spec Manager만 변경할 수 있다.

다른 Agent는 상태 변경 요청을 직접 수행하지 않고, 자신이 낸 결과와 추천만 기록한다.

Spec Manager의 처리 순서는 아래처럼 고정하는 편이 맞다.

1. Spec Validator 결과를 확인한다.
2. 처리 상태가 `완료`인지 확인한다.
3. 기계적 검토에서 오류가 없으면 상태 흐름표를 적용한다.
4. 다음 상태를 반영한다.
5. 상태 전이 로그를 남긴다.

상태 변경은 deterministic rule engine을 통해 처리하는 것을 기본안으로 둔다.

## 5.7 검토 요청

검토 요청은 단순 코멘트가 아니라 별도 운영 객체로 다룬다.

검토 요청은 아래 목적을 가진다.

- 사용자 판단이 필요한 지점을 명확히 드러낸다.
- 판단에 필요한 근거를 한곳에 모은다.
- 사용자가 선택 가능한 대안 2~3개를 비교 가능하게 제시한다.
- 응답 이후 Spec Validator가 다시 판정할 수 있게 구조화된 입력을 남긴다.

최소 필드는 아래가 필요하다.

- review request ID
- 대상 spec ID
- 생성 시각
- 생성 주체
- 검토가 필요한 이유
- 판단 근거 요약
- 사용자가 답해야 할 질문
- 선택 가능한 대안 2~3개
- 사용자 응답
- 사용자 응답 타입
- 응답 이후 Spec Validator 판정 결과
- 종료 여부

검토 요청은 열려 있는 동안 해당 스펙의 처리 상태를 `사용자검토`로 유지하는 근거가 된다.

사용자 응답 모델은 최소한 아래 3가지를 지원하는 편이 맞다.

- 대안 선택 후 승인
- 추가 코멘트와 함께 반려
- 부분 수정 후 승인

`부분 수정 후 승인`은 사용자가 에이전트 제안의 일부를 직접 고치고 승인하는 루프다. 이 경우 수정된 내용과 차이를 Spec Validator가 다시 읽고, 후속 ambiguity가 없으면 처리 상태를 `완료`로 만들고, 남아 있으면 새 검토 요청을 만든다.

# 6. 사용자와 웹 서비스

## 6.1 사용자 역할

- Agent가 만든 스펙을 검토한다.
- 직접 스펙을 수정할 수 있다.
- Planner에게 수정 요청을 줄 수 있다.
- 사용자 테스트를 수행하고 결과를 남긴다.
- 최종 승인 또는 반려 판단에 참여한다.

## 6.2 웹 서비스 구성

웹 서비스는 최소 네 축이 필요하다.

- 스펙 편집기: JSON 스펙을 사람이 읽기 쉬운 문서 형태와 폼 형태로 보여주고 수정
- 전체 스펙 뷰어: 프로젝트 전체 스펙을 문서 형태로 탐색
- 스펙 칸반 보드: 현재 상태와 병목, 검토 필요 항목을 시각화
- 검토 요청함: Spec Validator가 올린 검토 요청을 목록과 상세로 조회하고 응답

추가로 있으면 좋은 화면:

- 스펙 상세 타임라인
- 테스트 증거 뷰어
- 질문/답변 스레드 화면
- 검토 요청 상세 화면
- 프로젝트별 Agent 실행 현황 대시보드
- assignment 현황 화면

검토 요청함은 별도 영역으로 두는 편이 맞다.

- 목록에서는 미해결 검토 요청을 우선 보여준다.
- 상세에서는 검토 사유, 근거, 질문, 대안, 관련 테스트 결과, 관련 증거를 함께 보여준다.
- 사용자가 대안을 선택하거나 의견을 남기거나 부분 수정 후 승인하면 Spec Validator가 다시 확인하고, 모호함이 없으면 처리 상태를 `완료`로 변경한다.
- 모호함이 남아 있으면 같은 스펙에 새 검토 요청을 생성한다.

# 7. 스펙 모델

스펙은 단순 문서가 아니라, Agent가 읽고 상태를 바꾸고 테스트를 연결하는 운영 객체여야 한다.

## 7.1 기본 필드

- 정의
- problem
- goal
- acceptance criteria
- 테스트
- 상태
- 처리 상태
- 타입
- risk level
- 의존성
- assignments
- 활동 내역
- 검토 요청

## 7.2 테스트

테스트는 아래 세 종류를 기본으로 둔다.

- 단위 테스트
- E2E 테스트
- 사용자 테스트

각 테스트는 가능하면 아래 정보를 가져야 한다.

- 테스트 ID
- 테스트 타입
- 검증 대상 acceptance criteria ID
- 준비 절차
- 실행 절차
- 기대 결과
- 최근 실행 결과
- 최근 증거 경로

사용자 테스트는 아래 정책을 따른다.

- Pass/Fail 응답은 필수
- 코멘트는 옵션
- 스크린샷은 옵션
- 테스트 케이스 단위로 체크한다

## 7.3 Assignment

실제 운영에는 "어떤 Agent가 현재 이 스펙을 맡고 있는가"가 필요하다. 스펙 상태만으로는 concurrency control이 불가능하다.

권장 구조:

```text
spec
   |
   assignments
      |
      assignment
         id
         agent
         type
         status
         startedAt
         finishedAt
         result
```

assignment 최소 필드는 아래가 필요하다.

- assignment ID
- spec ID
- agent ID 또는 agent role
- assignment type
- status: `queued | running | completed | failed | cancelled`
- startedAt
- lastHeartbeatAt
- timeoutAt 또는 timeoutSeconds
- finishedAt
- result summary
- related worktree

예시:

- assignment(agent: developer-a, type: implementation, status: running)
- assignment(agent: test-validator, type: validation, status: queued)

이 모델이 있어야 아래가 가능하다.

- agent concurrency control
- 현재 작업자 표시
- 실패한 assignment 재시도 추적
- 동일 스펙에 대한 중복 실행 방지
- stale assignment 감지와 lock 회수

운영 안전장치도 필요하다.

- `running` assignment는 heartbeat를 주기적으로 갱신해야 한다.
- heartbeat가 timeout을 넘기면 Spec Manager가 stale assignment로 판정한다.
- stale assignment는 `failed` 또는 timeout 사유와 함께 종료 처리하고 관련 lock과 worktree 점유를 해제한다.
- 자동 회수 후에는 Planner 또는 사용자에게 재시도 여부를 검토 요청하는 편이 안전하다.

## 7.4 상태

현재 구상한 상태는 아래와 같다.

- 초안
- 대기
- 구현 검토
- 구현
- 테스트 검증
- 검토
- 활성
- 실패
- 완료

추가 원칙:

- 상태 변경은 Spec Manager만 수행할 수 있다.
- Developer, Test Validator, 사용자, 웹 UI는 상태 변경 요청에 필요한 데이터만 남긴다.
- 실제 상태 반영은 항상 Spec Manager를 통해서만 이뤄진다.

기존 runner 문서와는 차이가 있지만, 이번 방향은 새 구조를 기준으로 구현을 새로 작성하는 쪽이므로 상태 모델도 이 문서 기준으로 다시 고정하는 편이 맞다.

상태명은 UI 별칭이 아니라 저장되는 실제 값이어야 한다.

여기서 `검토`는 단순 코멘트 상태가 아니라, 미해결 review request가 하나 이상 열려 있는 운영 상태를 의미한다.

여기서 `구현 검토`는 Architect가 구현 착수 전 구조, 경계, 접근 방법을 검토하는 상태를 의미한다.

기본 흐름은 `대기 -> 구현 검토 -> 구현 -> 테스트 검증 -> 검토 또는 활성/완료`를 기준으로 잡는다.

## 7.5 처리 상태

`처리 상태`는 담당자 작업과 검증 루프를 추적하는 별도 축으로 유지한다.

권장 정의:

- `상태`: 비즈니스 흐름상 현재 단계
- `처리 상태`: 해당 단계에서 담당자의 작업, 검증, 사용자 검토가 어느 단계에 있는지 나타내는 값

예시:

- 상태 = `구현`, 처리 상태 = `처리중`
- 상태 = `검토`, 처리 상태 = `사용자검토`

처리 상태 후보:

- 대기
- 처리중
- 검토
- 사용자검토
- 완료
- 실패
- 보류

처리 상태 규칙은 아래처럼 고정한다.

- 처리 상태가 `대기`인 스펙을 각 담당자가 가져가 작업을 시작하면 `처리중`으로 바꾼다.
- 담당자 작업이 끝나면 처리 상태를 `검토`로 바꾼다.
- Spec Validator가 사용자 검토가 필요하다고 판단하면 검토 의견을 만들고 `사용자검토`로 바꾼다.
- 사용자가 검토를 완료하면 처리 상태는 다시 `검토`로 돌아간다.
- Spec Validator가 재검토 후 충분하다고 판단하면 `완료`로 바꾼다.
- 불충분하면 `대기`로 되돌려 재작업하게 한다.
- 이 루프가 최대 3회 반복되어도 수렴하지 않으면 `실패`로 바꾼다.
- `보류`는 외부 의존성, 환경 문제, 정책 대기처럼 당장 진행할 수 없는 경우에 사용한다.

## 7.6 타입

- 기능
- 태스크

타입은 아래처럼 해석한다.

- 기능: 사용자나 시스템의 지속되는 계약을 만든다.
- 태스크: 구현, 마이그레이션, 정리 같은 수행 작업이다.

## 7.7 의존성

- depends on
- blocks

추가로 고려할 관계:

- duplicates
- related to
- discovered from

의존성 실패의 폭포수 효과도 운영 규칙으로 고정하는 편이 안전하다.

- blocker spec이 최종 `실패`가 되면 해당 spec에 `depends on`으로 연결된 종속 spec을 검사한다.
- 종속 spec이 `대기`, `처리중`, `검토`, `사용자검토` 중 하나면 Spec Manager가 처리 상태를 `보류`로 전환한다.
- 이때 활동 로그에 upstream dependency failure 사유를 남기고 Planner와 사용자에게 검토 요청을 생성한다.
- 검토 요청에서는 우회 구현, 스펙 분할, blocker 복구 대기, 종속 spec 폐기 중 어떤 경로를 택할지 결정하게 한다.
- 종속 spec이 이미 `완료` 또는 `실패`라면 추가 전파는 하지 않는다.

## 7.8 활동 내역

최소 필드는 아래가 필요하다.

- timestamp
- agent
- action
- message
- artifact

추가 권장 필드:

- model
- status transition
- result type
- related acceptance criteria IDs
- related test IDs

활동 이력은 `spec.json` 안에 누적하지 않고 `activity/*.jsonl`에 분리 저장하는 편이 맞다.

- `spec.json`에는 현재 상태와 핵심 메타데이터만 유지한다.
- 타임라인, 판정 로그, 실행 로그는 append-only `jsonl` 파일에 누적한다.
- 웹 UI는 이 로그를 읽어 타임라인을 렌더링한다.
- Agent도 최근 필요한 구간만 선택적으로 읽을 수 있다.

# 8. 지금 보강해야 할 부분

현재 메모에서 특히 보강이 필요한 지점은 아래와 같다.

## 8.1 상태 전이 규칙

가장 중요하다. 특히 "누가 상태를 바꿀 수 있는가"를 먼저 잠가야 한다.

현재 방향의 핵심 규칙은 아래다.

- 스펙 상태는 Spec Manager만 변경할 수 있다.
- 다른 Agent는 결과, 근거, 추천만 남긴다.
- 사용자는 검토 요청과 사용자 테스트에 응답하지만 상태를 직접 바꾸지 않는다.

필수 질문:

- Planner가 스펙 초안을 만들면 Spec Manager는 어떤 조건에서 `대기`로 올리는가?
- Planner 초안의 acceptance criteria가 테스트 가능하지 않을 때 Architect는 어떤 기준으로 반려하고, 어떤 조건에서 다시 제출하게 하는가?
- `대기` 상태의 스펙을 어떤 조건에서 `구현 검토`로 올리고, Architect 검토 통과 시 어떤 조건에서 `구현`으로 넘기는가?
- Developer가 남긴 결과를 Spec Manager는 어떤 조건에서 `테스트 검증` 후보로 인정하는가?
- Test Validator의 최종 합격 판정을 Spec Manager는 어떤 규칙으로 상태에 반영하는가?
- Spec Validator는 어떤 조건에서 `검토`, `사용자검토`, `완료`, `실패`를 판정하는가?
- 사용자 검토가 필요한 상황을 Spec Validator가 어떤 규칙으로 검토 요청으로 승격하는가?
- 실패 시에는 어떤 상태로 되돌아가고, 재시도와 검토 요청을 Spec Validator가 어떻게 구분하는가?
- Developer와 Test Validator가 충돌할 때 Spec Validator는 retry loop를 몇 번까지 허용하고, 언제 사람 개입으로 승격하는가?
- blocker spec이 `실패`일 때 종속 spec을 어떤 범위까지 `보류`로 전파할지, 그리고 누가 재개를 승인하는가?
- heartbeat가 끊긴 assignment를 언제 stale로 판정하고, 자동 회수 뒤 어떤 재시도 정책을 적용할 것인가?

## 8.2 Authoritative source

이 부분은 이제 명확히 고정하는 편이 맞다.

- 구조화된 JSON 스펙이 오리진이다.
- 사람이 읽는 문서 화면은 JSON을 렌더링한 결과다.
- 웹 서비스 DB는 캐시나 인덱스 용도로 사용할 수는 있지만 오리진이 되어서는 안 된다.

권장 구조:

- `spec.json`: 스펙 본문, acceptance criteria, 테스트, 상태, assignment, 검토 요청 참조를 포함한 authoritative file
- `activity/*.jsonl`: 실행 로그와 판정 로그
- `evidence/`: shared home 아래에 저장되는 테스트 증거
- `artifacts/`: 실행 중 산출물
- `worktree/metadata.json`: 연결된 워크트리 정보

이렇게 두면 웹 UI, CLI, Runner가 모두 같은 원본을 읽게 된다.

## 8.3 프로젝트 식별과 경로 해석

멀티 프로젝트로 가면 아래 문제가 바로 생긴다.

- 동일한 폴더명이 다른 저장소에서 충돌할 수 있다.
- 로컬 경로가 바뀌면 기존 스펙 경로가 깨질 수 있다.
- VS Code 확장, CLI, Runner가 같은 프로젝트를 같은 방식으로 식별해야 한다.

프로젝트 식별 방식은 아래처럼 고정한다.

- Flow가 발급한 영구 project ID를 기본 식별자로 사용한다.
- `config.yaml`에 root path, remote URL, display name, aliases를 저장한다.
- 최초 등록 시에는 git remote URL과 default branch를 보조 신호로 사용한다.
- git remote가 없는 로컬 프로젝트는 정규화된 절대 경로 해시를 bootstrap 힌트로 사용한다.
- bootstrap 이후에는 발급된 영구 project ID를 사용하므로 경로 해시는 식별 보조값으로만 남긴다.

즉 "영구 project ID + path aliases + remote metadata" 조합을 기본안으로 확정한다.

## 8.4 동시 실행과 락

초기 단계에서는 복잡한 락 시스템보다 운영 규칙을 강하게 두는 편이 낫다.

현재 전제는 다음과 같다.

- 작업은 워크트리 기반으로 수행한다.
- 처음에는 Agent마다 동시에 하나의 작업만 맡는다.

이 전제에서 필요한 규칙은 아래 정도면 충분하다.

- 같은 스펙 동시 수정
- 같은 브랜치 동시 사용
- 로그 append 충돌
- 상태 덮어쓰기

- 스펙마다 활성 worktree를 최대 1개만 둔다.
- Agent마다 활성 assignment를 최대 1개만 둔다.
- Spec Manager가 worktree 할당 전에 spec lock과 agent lock을 확인한다.
- assignment가 `running`인 Agent에는 새 assignment를 만들지 않는다.
- assignment는 heartbeat와 timeout을 가져야 하고, 일정 시간 heartbeat가 없으면 stale로 회수한다.
- lock 정보는 파일 또는 작은 JSON 메타데이터로 시작하고, 나중에 필요하면 DB 락으로 확장한다.
- 같은 스펙을 다시 작업해야 하면 기존 worktree를 재사용하거나 명시적으로 정리 후 재생성한다.

deadlock 방지 규칙도 같이 두는 편이 좋다.

- Spec Manager가 주기적으로 `running` assignment의 heartbeat를 검사한다.
- stale assignment를 발견하면 해당 assignment를 실패 종료하고 spec lock, agent lock, worktree lock을 해제한다.
- timeout 회수는 활동 로그와 검토 요청에 남긴다.
- 동일 assignment를 자동 재실행할지, Planner 재판단으로 되돌릴지는 assignment type과 최근 실패 횟수에 따라 구분한다.

## 8.5 테스트 결과 판정 책임

현재 구상은 역할 분리가 좋지만, 아래 경계는 더 선명해야 한다.

- 테스트 실행 책임자: Developer
- 테스트 적합성 판단 책임자: Test Validator
- 최종 합격 판정 책임자: Test Validator

즉 구현한 곳에서 테스트를 실행하고, 그 테스트가 적절한지와 최종 합격 여부는 Test Validator가 판단한다. Spec Manager는 그 판정 결과와 다른 운영 신호를 보고 상태만 반영한다.

## 8.6 사용자 테스트 UX

사용자 테스트는 반드시 구조화되어야 한다.

- Pass/Fail은 필수
- 코멘트는 옵션
- 스크린샷 업로드는 옵션
- 테스트 케이스 단위 체크

권장 데이터 형태:

- test ID
- result: `pass | fail`
- comment?: string
- screenshotPaths?: string[]
- submittedBy
- submittedAt

이 정도면 초기 운영에는 충분하다.

## 8.7 워크트리에 스펙을 어떻게 전달할 것인가

이건 실제 개발 경험에 큰 영향을 주기 때문에 초기에 방향을 정하는 편이 좋다.

워크트리 전달 방식은 아래처럼 고정한다.

- authoritative source는 shared home의 spec JSON 하나만 둔다.
- 워크트리에는 `flow.exe`와 짧은 부트스트랩 문서만 둔다.
- 부트스트랩 문서에는 `flow spec show`, `flow spec tests`, `flow evidence add` 같은 핵심 명령만 안내한다.
- 필요 시 `flow spec export --summary` 같은 명령으로 읽기 전용 요약 문서를 생성할 수 있게 한다.

즉 워크트리에는 원본 스펙 복사본을 두지 않는다.

## 8.8 마이그레이션 전략

이번 방향은 기존 코드를 이어서 손보는 것보다, 기존 코드는 참조만 하고 새 구조로 다시 작성하는 쪽이 더 맞다.

권장 순서:

1. 상태 모델과 저장 구조를 먼저 확정한다.
2. JSON 스펙 스키마와 검토 요청 스키마를 먼저 고정한다.
3. assignment 모델과 concurrency rule, timeout 회수 규칙을 먼저 고정한다.
4. Planner 초안에 대한 acceptance criteria 테스트 가능성 프리패스 규칙을 고정한다.
5. Spec Validator, Spec Manager, Architect, Developer, Test Validator의 입력/출력 계약을 정의한다.
6. 워크트리 부트스트랩 방식과 `flow.exe` 계약을 정의한다.
7. blocker 실패 시 종속 spec `보류` 전파 규칙을 상태 전이표에 포함한다.
8. 그 다음 Runner, 웹 편집기, 칸반, 검토 요청함을 새로 구현한다.

# 9. 권장 개발 원칙

- 상태 모델을 먼저 고정하고 구현에 들어간다.
- UI보다 저장 구조와 상태 전이 규칙을 먼저 정한다.
- Agent 책임은 최대한 겹치지 않게 만든다.
- 모든 판단은 acceptance criteria, 테스트, 증거에 연결한다.
- 자동화 실패는 재시도 가능 실패와 사람 판단이 필요한 실패로 나눈다.
- 기존 코드는 참조만 하고 새 구조로 다시 작성한다.
- 기존 runtime과 문서는 마이그레이션 대상이 아니라 참고 자료로 취급한다.

# 10. 권장 후속 작업

우선순위가 높은 후속 작업은 아래와 같다.

1. 상태 모델 통합안 작성
2. JSON 스펙 스키마와 review request 스키마 확정
3. assignment 스키마, heartbeat, timeout 회수 규칙 확정
4. Spec Validator 처리 상태 규칙표와 Spec Manager 상태 전이 규칙표 작성
5. dependency failure cascade와 `보류` 전파 규칙 정의
6. Spec Validator의 AC 테스트 가능성 프리패스 체크리스트 정의
7. Agent별 입력/출력 계약 정의
8. 워크트리 부트스트랩 규격과 `flow.exe` 명령 계약 정의
9. 웹 UI의 검토 요청함 목록/상세/응답 흐름 설계

# 11. 한 줄 결론

방향 자체는 맞다. 다만 지금 필요한 것은 "AI Agent를 도입한다"는 선언보다, JSON 스펙 오리진, Spec Validator와 Spec Manager의 역할 분리, 검토 요청 모델, 워크트리 전달 계약을 먼저 고정하는 일이다. 그 기준이 정리되면 Runner, 웹 UI, CLI, Agent 구현은 그 위에 일관되게 쌓을 수 있다.
