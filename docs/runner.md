칸반 보드
웹으로 구현

스펙 및 관련 파일 위치: `~/.flow/specs/<폴더명>/<스펙ID>/`
스펙 본문은 마크다운 파일을 기준으로 한다.
Copilot ACP 모드를 사용한다.
모든 실행 이력, 판단 근거, 테스트 결과, 질문, 사용자 응답은 스펙 위치에 로그로 남긴다.

# 목표

이 문서는 Runner가 스펙을 자동으로 처리할 때 필요한 최소 운영 규칙을 정의한다.

- 웹 칸반 보드에서 스펙 상태를 한눈에 볼 수 있어야 한다.
- 자동 처리와 사용자 개입의 경계를 명확히 해야 한다.
- 모든 판단 근거는 스펙 폴더 안의 파일과 로그로 추적 가능해야 한다.
- 재시도, 실패, 사용자 질문, 사용자 테스트도 모두 이력으로 남아야 한다.

# 스펙

## 스펙 타입

- 기능
- 태스크

기능은 최종 상태가 `활성`이고, 태스크는 최종 상태가 `완료`다.

## 스펙 저장 구조

각 스펙은 독립 폴더를 갖는다.

```text
~/.flow/specs/<폴더명>/<스펙ID>/
	spec.md
	meta.json
	activity.log.md
	questions.md
	tests/
		unit/
		e2e/
		user/
	evidence/
		unit/
		e2e/
		user/
	artifacts/
	worktree/
```

- `spec.md`: 사람이 읽는 요구사항, 조건, 테스트 정의, 진행 메모
- `meta.json`: 상태, 타입, 작업 횟수, 마지막 오류, 재시도 시각 등 구조화 데이터
- `activity.log.md`: append-only 실행 로그
- `questions.md`: 사용자 질문, 답변 상태, 검토 메모
- `tests/`: 테스트 정의 또는 테스트 케이스 참조
- `evidence/`: 테스트 증거 저장 위치
- `artifacts/`: 실행 중 생성된 산출물
- `worktree/`: 연결된 워크트리 경로, 브랜치명, 정리 상태 같은 메타 정보

## 스펙 본문 필수 항목

최소한 아래 항목은 `spec.md`에 있어야 한다.

- 제목
- 스펙 ID
- 스펙 타입
- 배경과 목적
- 범위
- 제외 범위
- 조건 목록
- 테스트 정의
- 의존성 또는 선행 조건
- 사용자에게 물어볼 질문

## 조건

스펙은 여러 조건을 가진다. 조건은 최종 판정의 최소 단위다.

각 조건은 아래를 가진다.

- 조건 ID
- 설명
- 연결된 테스트 목록
- 현재 상태: `초안`, `검토 필요`, `검증 완료`
- 마지막 판정 근거

조건 단위로 어떤 테스트가 무엇을 검증하는지 연결되어야 한다.

## 스펙 상태

스펙 상태는 화면에 보이는 값과 저장되는 값이 같다.

아래 상태 이름을 그대로 저장하고 그대로 표시한다.

- `초안`
- `대기`
- `작업`
- `테스트 검증`
- `리뷰`
- `검토`
- `활성`
- `완료`

즉 `작업`, `테스트 검증`, `리뷰`는 단순한 UI 별칭이 아니라 실제 스펙 상태다.

별도 내부 메타데이터가 필요하더라도 그것은 보조 정보일 뿐이며, 스펙의 authoritative 상태는 항상 위 상태 값 중 하나여야 한다.

## 테스트

### 스펙은 테스트를 여럿 가진다.

- 테스트 타입: 단위 테스트, E2E 테스트, 사용자 테스트
- 사용자 테스트는 실제 사람이 직접 테스트하고 결과를 입력하는 방식이다.

### 테스트 정의 필드

각 테스트는 아래 정보를 가진다.

- 테스트 ID
- 테스트 타입
- 검증 대상 조건 ID
- 준비 절차
- 실행 절차
- 기대 결과
- 증거 타입
- 최근 실행 결과
- 최근 증거 경로

### 테스트 증거

- 단위 테스트는 입력값과 출력값, 실행 로그를 증거로 남긴다.
- E2E 테스트는 이미지, 필요하면 동영상, 실행 로그를 증거로 남긴다.
- 사용자 테스트는 사용자의 응답, 체크 결과, 코멘트를 증거로 남긴다.

### 증거 저장 규칙

- 모든 증거는 스펙 폴더 내부에만 저장한다.
- 증거 파일명에는 시각과 테스트 ID를 포함한다.
- 테스트 결과 요약은 `activity.log.md`에도 남긴다.
- E2E 이미지 검토 시에는 VLM 판단 결과와 함께 실패 가능성이 높은 3가지 이유를 기록한다.

## 작업 횟수

- `작업` 상태에서 실제 구현 시도를 시작할 때마다 작업 횟수를 1 증가시킨다.
- 작업 횟수가 3 이상이 되면 해당 스펙은 자동으로 `검토` 상태로 보내 사용자 또는 운영자 판단을 받는다.
- `검토` 상태에서 나와 다시 `대기`로 돌아갈 때 작업 횟수를 초기화한다.
- 단순 조회, 로그 수집, 상태 판정만 한 경우는 작업 횟수에 포함하지 않는다.

# Runner

Runner는 무한 루프를 돌면서 스펙 상태를 변경한다.

## 기본 원칙

- 한 번의 사이클에서 구현, 테스트 검증, 리뷰를 순차적으로 처리한다.
- 자동 파이프라인이 끝나기 전까지는 사용자에게 불필요하게 상태를 노출하지 않는다.
- 정말로 사용자 행동이 필요할 때만 `검토` 상태로 보낸다.
- 자동 재시도가 필요한 경우는 `대기`로 되돌린다.
- 모든 상태 전이에는 이유와 근거 파일 경로를 남긴다.

## 워크트리 규칙

- `대기` 상태의 스펙을 집기 전에 기존 워크트리 메타데이터를 정리한다.
- 스펙별 브랜치와 워크트리를 만든다.
- 이미 워크트리가 있다면 default branch 기준으로 rebase 하거나 재생성한다.
- stale worktree 정보가 남아 있으면 prune 후 다시 생성한다.
- `활성` 또는 `완료`로 확정되면 워크트리를 정리한다.
- 실패로 `대기`로 되돌아가더라도 재시도에 필요한 정보는 스펙 로그에 남긴다.

## 로그 규칙

`activity.log.md`는 append-only로 유지한다.

각 로그 항목은 최소한 아래를 포함한다.

- 시각
- 역할
- 실행 주체
- 사용 모델
- 요약
- 상세 코멘트
- 관련 파일 경로
- 상태 변경
- 결과 분류

결과 분류는 아래 집합을 기본으로 한다.

- `handoff`
- `requeue`
- `needs-review`
- `verified`
- `done`
- `blocked`

## 재시도와 쿨다운

- 네트워크 오류, rate limit, 실행 환경 문제처럼 일시적 실패는 즉시 무한 재시도하지 않는다.
- `meta.json`에 다음 재시도 가능 시각을 기록한다.
- 쿨다운 중인 스펙은 `대기` 컬럼에 남기되, 차단 사유를 UI에서 보여준다.
- 작업 중 프로세스가 비정상 종료되면 crash recovery 로그를 남기고 `대기`로 복귀시킨다.

## 상태별 처리

### 초안

- 사용자나 Spec Planner agent가 스펙을 추가한다.
- 이 단계에서는 구조 보강, 조건 정리, 테스트 정의 보강만 한다.
- 충분히 다듬어진 후 사용자가 `대기`로 옮긴다.

### 대기

- 자동 처리 대상 큐다.
- 다른 상태에서 문제가 있어서 재작업이 필요한 경우에도 여기로 돌아온다.
- open question이 모두 해결되었더라도 Runner가 다시 판정하기 전에는 바로 최종 상태로 가지 않는다.

### 작업 (Developer Copilot `gpt-5.4`)

- 워크트리를 생성한다. 이미 있다면 default branch 기준으로 동기화한다.
- `대기` 상태의 스펙을 하나 가져와 해결한다.
- 테스트를 먼저 만들거나 보강한 뒤 구현한다.
- 구현 중 필요한 로그, 산출물, 테스트 결과를 스펙 폴더에 저장한다.
- 테스트 통과에 실패했다면 무엇을 시도했고 어떻게 실패했는지 기록한다.
- 구현 중 사용자 의견이 꼭 필요하면 질문을 스펙에 추가하되, 최종 상태 전환은 리뷰 단계에서 한다.
- 구현이 끝나면 스펙 상태를 `테스트 검증`으로 변경한다.

주의:

- Developer는 최종 성공 여부를 확정하지 않는다.
- Developer 종료만으로 `검토` 상태로 보내지 않는다.

### 테스트 검증 (Test Validator Copilot `gpt-5-mini`)

- 구현된 테스트가 스펙과 테스트 정의에 맞는지 확인한다.
- 테스트가 조건을 실제로 검증하는지 본다.
- 너무 사소하거나, 비결정적이거나, 결과 해석이 잘못된 테스트를 걸러낸다.

판정 규칙:

- 적절하면 스펙 상태를 `리뷰`로 변경한다.
- 부적절하면 어떤 부분이 잘못되었는지 스펙에 첨부하고 `대기` 상태로 되돌린다.

### 리뷰 (Test Reviewer Copilot `gpt-5-mini`)

- 테스트 코드와 테스트 증거를 보고 각 테스트의 성공/실패를 판정한다.
- 증거와 실행 로그를 함께 확인한다.
- 이미지의 경우 VLM을 사용한다.
- VLM에게 이미지를 줄 때 이 테스트가 실패했을 가능성이 있는 3가지 이유를 찾도록 해서 검증 결과를 더 신뢰할 수 있게 한다.
- 조건 단위로 `검증 완료`, `검토 필요`, `초안` 상태를 갱신한다.

판정 규칙:

- 사용자 테스트나 사용자 의견이 필요하면 `검토` 상태로 변경한다.
- 테스트 모두 성공인 경우 스펙 타입이 기능이면 `활성`, 태스크라면 `완료` 상태로 변경한다.
- 실패한 테스트가 있다면 개선 방향을 기록하고 `대기` 상태로 변경한다.
- 증거 부족, 실행 불가, 환경 오류도 `검토`가 아니라 기본적으로 `대기`로 보낸다.

### 검토

`검토`는 자동 파이프라인이 끝났고 이제 사용자 행동이 필요한 상태를 의미한다.

이 상태로 가는 경우:

- 사용자 질문에 대한 답변이 필요함
- 사용자 테스트가 필요함
- 운영 판단 또는 최종 확인이 필요함

규칙:

- 사용자의 의견이 필요한 경우 스펙에 질문을 표시한다.
- 질문은 응답 대기, 응답 완료를 구분해서 기록한다.
- open question이 하나라도 남아 있으면 `검토`를 유지한다.
- 모든 질문에 답했더라도 바로 `활성`이나 `완료`로 바꾸지 않고 Runner가 다시 판정한다.
- 사용자 테스트는 `Pass`, `Fail` 두 상태를 가지며, `Fail`이면 의견을 추가할 수 있어야 한다.
- 사용자가 모든 사용자 테스트에 체크했고 `Fail`이 하나라도 있으면 스펙은 `대기`로 간다.
- 사용자 테스트까지 모두 통과하면 스펙 타입에 따라 `활성` 또는 `완료`로 변경한다.

### 활성 / 완료

- `활성`은 기능이 구현되고 검증까지 끝난 상태다.
- `완료`는 태스크가 끝난 상태다.
- 이 상태로 왔을 때 워크트리를 정리한다.
- 최종 판정 근거, 마지막 증거 경로, 관련 커밋 정보를 로그에 남긴다.

# 칸반 보드 규칙

웹 칸반 보드는 아래 컬럼을 가진다.

- 초안
- 대기
- 작업
- 테스트 검증
- 리뷰
- 검토
- 활성
- 완료

보드 카드에는 최소한 아래를 보여준다.

- 스펙 ID와 제목
- 스펙 타입
- 현재 상태
- 작업 횟수
- open question 수
- 테스트 수와 최근 결과
- 마지막 업데이트 시각
- 마지막 실패 사유 또는 검토 사유

## 상세 화면

스펙 상세 화면에는 아래가 보여야 한다.

- 조건 목록과 상태
- 테스트 목록과 최근 결과
- 증거 미리보기
- 리뷰 코멘트
- 사용자 질문과 답변 상태
- activity 로그
- 관련 파일 경로

조건, code references, 리뷰 로그는 접기/펼치기가 가능하면 좋다.

# 병렬 처리 메모

초기 구현은 단일 루프 기반으로 시작하되, 이후에는 아래 최적화를 고려할 수 있다.

- Developer가 한 스펙 구현을 끝내면 다음 스펙 구현으로 바로 넘어간다.
- 이전 스펙은 Test Validator와 Test Reviewer가 이어서 처리한다.
- 즉 `Developer -> Test Validator -> Tester`를 완전히 직렬로 한 스펙씩 처리하는 대신, stage별 파이프라인 중첩 실행을 허용할 수 있다.

다만 이 경우에도 아래는 유지해야 한다.

- 한 스펙 안에서는 stage 순서가 보장되어야 한다.
- 상태 전이와 로그 기록은 직렬화되어야 한다.
- 같은 스펙을 두 작업자가 동시에 수정하면 안 된다.

# 미정 항목

아래는 후속 설계가 더 필요한 항목이다.

- `spec.md`와 `meta.json` 사이의 authoritative source 규칙
- 사용자 테스트 입력 UI 상세 형식
- 질문 스레드의 닫힘 규칙
- 여러 개의 Runner 프로세스가 동시에 동작할 때 락 전략
- 증거 보존 기간과 정리 정책
